# JudgmentOS 5軸アライメント変更説明書

## 概要

シャープファイナンス向けヒアリングシステムにおいて、仕様書に定義した5軸フレームワーク（PBM・知識レイヤー・リスク軸・データ分類・メダリオン）に沿って、AIの動作ロジックを全面的にアップデートしました。

---

## 1. バイアス検出（認知補正）

### As-Is（変更前）

- AIが検出できるバイアスは **6種類**
  - 楽観的断定（「大丈夫」「問題ない」）
  - 確証バイアス（「前回もうまくいった」「いつも」）
  - サンクコスト（「ここまでやったから」「せっかく」）
  - **グループシンキング**（「みんなも」「普通は」「業界では当たり前」）
  - 過信（「絶対」「100%」「間違いない」）
  - **曖昧な根拠**（「なんとなく」「肌感覚」「直感」）
- バイアスを検出したら **毎回すべて** LLMに通知（回数制限なし）
- 補正質問の **具体例がない** ため、LLM任せで質問の質にばらつきあり

### To-Be（変更後）

- AIが検出できるバイアスは仕様書に準拠した **5種類** に変更
  - 楽観バイアス（「大丈夫」「問題ない」）
  - 確証バイアス（「前回もうまくいった」「いつも」）
  - サンクコスト（「ここまでやったから」「せっかく」）
  - **権威バイアス**（「上の方針」「部長が」「専門家が」）← 新規追加
  - 過信バイアス（「絶対」「100%」「間違いない」）
- グループシンキングと曖昧な根拠は削除（仕様書の設計方針として、与信審査で特に危険な5種類に絞り込み）
- **挿入タイミング制御を追加**（仕様書3.4.2節準拠）
  - 1回のヒアリングセッションで **最大2回まで** しか補正質問を挿入しない
  - 前回の補正質問から **最低3往復の間隔** を空ける
  - 検出信頼度が **0.7以上** のバイアスのみ対象とする
- 各バイアスに **ソクラテス式質問テンプレート** を追加
  - 楽観バイアス → 「もし最悪のケースが起きた場合、この見通しはどう変わりますか？」
  - 確証バイアス → 「逆に、この前提が当てはまらなかったケースはありますか？」
  - サンクコスト → 「もし今日が初日だとして、同じ判断をしますか？」
  - 権威バイアス → 「その方の意見を除外した場合、データや事実だけから同じ結論に至りますか？」
  - 過信バイアス → 「この確信度を下げる要因があるとすれば、何が考えられますか？」

### 変更の意図

仕様書で中島氏から出た要件「目立たず、自然に」を実現するため。以前は検出するたびに毎回LLMに通知していたため、補正質問が頻繁に挿入されて不自然になるリスクがあった。今回のタイミング制御により、自然な会話の流れの中で最大2回だけ、適切な間隔を空けて補正質問が入る設計になった。

## 2. リスク軸（Risk Axis）

### As-Is（変更前）

- ヒアリングで抽出した知識に **リスクの種類は記録されていなかった**
- 「どのリスクに関する情報か」が分類されず、後から分析しにくい状態

### To-Be（変更後）

- 抽出した知識ごとに **6つのリスク軸のいずれか** を自動分類して記録
  1. **対象そのもののリスク** — 借り手の信用力、事業の成立性など
  2. **構造・契約のリスク** — 取引構造、契約条件、担保に関するリスク
  3. **外生環境リスク** — マクロ経済、市場、規制環境のリスク
  4. **実行・運営リスク** — 運営体制、オペレーション上のリスク
  5. **集中・相関リスク** — ポートフォリオの集中、相関リスク
  6. **情報・前提リスク** — データの品質、前提条件の信頼性に関するリスク
- これはエクセルの判断フロー・個別判断表でラベリングされた「risk_axis」列と同じ分類体系

### 変更の意図

エクセルで定義済みの6軸リスク分類をシステム内の知識抽出にも適用することで、「このヒアリングではどのリスク軸の情報が十分に取れているか/不足しているか」を可視化・分析できるようにする。

## 3. データ4Type分類

### As-Is（変更前）

- データベースに `data_type` というカラムは存在していたが **「Phase2用」として未使用**
- ヒアリングで得た知識の **データ特性（変わりやすさ）** が分類されていなかった
- データの出所（誰が言ったか）や検証の必要性も記録なし

### To-Be（変更後）

- 抽出した知識ごとに以下の3つの観点で分類して記録

**データの変わりやすさ（4タイプ）：**
| 分類 | 意味 | 例 |
|------|------|-----|
| D1（不変に近い原則） | 法規制・根本ルールなど、ほぼ変わらない情報 | 「貸倒・法的手続きありは全てNG」 |
| D2（比較的安定した方針） | 社内方針・業界基準など、年に数回変わる程度 | 「自己資本比率30%以上」 |
| D3（頻繁に変わる前提） | 市場条件・現状データなど、都度変わる情報 | 「直近2年以内の遅延有無」 |
| D4（過去の判断経緯） | 過去の審査結果・先例・教訓 | 「前回の審査でこう判断した」 |

**データの出所：**
| 分類 | 意味 |
|------|------|
| self_report | 審査員本人の発言・意見 |
| internal_db | 社内データベース・ドキュメントからの参照 |
| derived | 他のデータから導出・計算された情報 |
| third_party_db | 外部・第三者機関からの情報 |

**検証の必要性：**
| 分類 | 意味 |
|------|------|
| required | 独立した情報源での検証が必須 |
| optional | 検証があると望ましいが必須ではない |
| not_required | 自明、または検証済み |

### 変更の意図

エクセルの判断フロー・個別判断表と同じ分類体系（`data_4type`、`data_source`、`verification_requirement` 列）をシステム内の知識抽出にも適用する。これにより「この情報はどの程度安定しているか」「出所は信頼できるか」「検証が必要か」が記録され、審査の質向上と監査可能性を担保する。

## 4. ソクラティックフィルタリング（レイヤーカバレッジ追跡）

### As-Is（変更前）

- ヒアリングで抽出した知識に5つの知識レイヤー（原則/判断基準/リスク構造/案件事実/判断プロセス）のラベルは付いていた
- しかし **「どのレイヤーの情報がどれだけ集まっているか」は追跡されていなかった**
- AIが質問を考える際に、**カバーされていないレイヤーを意識する仕組みがなかった**

### To-Be（変更後）

- ヒアリングの各応答後に **レイヤーごとの抽出件数を自動カウント** して記録
  - 例：Layer 0（原則）: 2件、Layer 1（判断基準）: 3件、Layer 2（リスク構造）: 0件 ...
- AIが次の質問を考える際に、**現在のレイヤーカバレッジ状況をリアルタイムで参照**
- **カバーが0件のレイヤーがあれば、そのレイヤーに関する質問を優先** するようAIに指示
- Step 2（定量・定性評価）終了前に、**各レイヤー最低2件の抽出** を目標とする
- 各レイヤーに対応した質問戦略をAIに提供
  - Layer 0（原則）→ 絶対に変わらないルールや原則について聞く
  - Layer 1（判断基準）→ 判断基準、閾値、評価基準について聞く
  - Layer 2（リスク構造）→ リスク要因とその関係性について聞く
  - Layer 3（案件事実）→ この案件固有の事実やデータについて聞く
  - Layer 4（判断プロセス）→ 判断の過程、暗黙の前提、トレードオフについて聞く

### 変更の意図

仕様書のソクラティックフィルタリングの概念を実装。5つのレイヤーを「横断的に漏れなく聞く」ことで、ヒアリングの網羅性を担保する。以前はAIが自由に質問を選んでいたため、特定のレイヤー（特にLayer 0の原則やLayer 2のリスク構造）が聞き漏れるリスクがあった。

## 5. AIへの指示（プロンプト）更新

### As-Is（変更前）

- AIへの指示には知識レイヤー（0-4）の定義のみ記載
- 抽出する情報のフォーマットは `layer`、`content`、`hypothesis` の3項目のみ
- バイアス補正の具体的なルール（回数制限、間隔）なし

### To-Be（変更後）

- AIへの指示に以下を追加
  - **6つのリスク軸の定義と分類ガイド**
  - **4つのデータタイプの定義と分類ガイド**
  - **データソースと検証要件の分類ガイド**
  - **バイアス補正ルール**（1回の応答で1つまで、ソクラテス式質問テンプレート使用）
  - **クロスレイヤーカバレッジガイダンス**（未カバーレイヤー優先、各レイヤー最低2件目標）
- 抽出する情報のフォーマットを7項目に拡張
  - `layer`（知識レイヤー 0-4）
  - `content`（抽出された知識）
  - `hypothesis`（仮説）
  - `risk_axis`（リスク軸）← 新規
  - `data_4type`（データ変動性）← 新規
  - `data_source`（データ出所）← 新規
  - `verification_requirement`（検証要件）← 新規

### 変更の意図

エクセルで20列にわたって定義された5軸のラベリング体系を、AIの知識抽出プロセスにも適用する。AIがヒアリング中に抽出する知識が、エクセルのラベリングと同じ軸で自動分類されるようになる。

## 6. PBMステップ（変更なし）

| ステップ | 名称 | 変更 |
|---------|------|------|
| Step 0 | Hard NG排除 | 変更なし |
| Step 1 | 前提確認 | 変更なし |
| Step 2 | 定量・定性評価 | 変更なし |

PBMの3ステップ構造はそのまま維持。ステップ遷移のロジック（Step 0: dealbreaker確認 → Step 1: 前提3つ以上 → Step 2: カバレッジ0.8以上）も変更なし。

---

## 7. メダリオンアーキテクチャ（変更なし）

データベースに `medallion` カラム（bronze/silver/gold）は既に存在するが、シャープファイナンスでは現時点で適用不可のため、今回は変更なし。

## データベース変更（新規カラム）

| テーブル | 追加カラム | 用途 |
|---------|-----------|------|
| hearing_extracts | risk_axis | 抽出知識のリスク軸分類 |
| hearing_extracts | data_source | 抽出知識のデータ出所 |
| hearing_extracts | verification_requirement | 抽出知識の検証要件 |
| hearing_step_states | layer_coverage | レイヤーごとの抽出件数（カバレッジ追跡用） |
| bias_flags | confidence_score | バイアス検出の信頼度スコア |

既存の `hearing_extracts.data_type` カラム（Phase2用として未使用だったもの）は、今回から正式にD1-D4の値で使用開始。
