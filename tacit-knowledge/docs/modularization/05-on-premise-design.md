# オンプレミス提供方針

## 提供形態

エンプラ向けSaaSではデータ保有の観点からオンプレ or 先方サーバー管理のニーズが高い。
特に業務のコアデータを扱うため、以下3つの提供形態を想定する。

| # | 形態 | 説明 |
|---|------|------|
| 1 | SaaS版 | 標準のクラウド提供 |
| 2 | オンプレ: データ分離用途のみ | SaaS版と同一機能。データの保管場所が異なるだけ |
| 3 | オンプレ: データ分離 + カスタム用途 | feature_management BCでカバーできないコア部分のカスタマイズを含む |

## 方針: 形態1, 2（SaaS版 + データ分離オンプレ）

1, 2は同一リポジトリで管理可能。2が1に追いつく際の運用体制が重要。

### 決めるべきこと

- [ ] どこまでインフラ側やCIで自動化するか
- [ ] migrationやrakeタスク実行のフロー（忘れず行う仕組み）
- [ ] 特定のサーバーに対して実行できるCI（インフラ知見なしで運用可能に）
- [ ] リリース時に必要な作業内容のメモ運用（特にrakeタスク）
- [ ] 2→1（もしくは特定バージョン）に追従する際、作業内容の差分が見える仕組み

### 論点

コードのモジュール化設計部分が主な論点。Packwerkによるパック境界が明確であれば、オンプレ版でも同じパック構造を利用でき、デプロイ対象のパック選択で差異を吸収できる可能性がある。

## 方針: 形態3（カスタム用途）

feature_management BCでカバーできないコア部分のカスタマイズを想定する必要がある。
ただし需要の大部分は形態1, 2で占められる見込み。

形態3が必要になった際にもコンテキストが分かれていれば対応可能。モジュール化設計でBC間の境界を明確にしておくことが前提条件。

→ 形態3の詳細設計は需要が具体化した時点で検討する。
